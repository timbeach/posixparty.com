What to watch for: mouth shape on the consonants, eye contact, micro-head movements driven by audio energy. Identity should stay locked from the bible portrait — no drift. EchoMimic tends to look more pristine but with smaller motion range; Wan2.2-S2V-14B has noticeably more head/body motion and the mouth opens wider on vowels. Lower resolution on Wan2.2 is the default size mismatch (square input → 576×576 default); we can request 832×480 or 1280×720 explicitly for higher-fidelity outputs at the cost of more inference time.
Caveats: this is a single 4-second clip per model on the same input. Not a real evaluation yet — just first-light. Real evaluation needs 5-10 clips per model across the bible (different expressions, different audio energies) before we can call a winner. Both videos are floating busts against the bible-image background, not character-in-room compositions — same POC framing as the May-5 demos.

4. infrastructure rebuild — one command

The whole point of today's work: from "I provisioned a fresh H100 pod" to "the entire model stack is installed, the buckets are wired up, and FLUX/PuLID smoke-tests pass" should take one command and ~50 minutes of unsupervised wall-clock.

how the install runs

Single bash entry point with idempotent steps and a hard requirement on $HF_TOKEN:

  • scripts/install_h100_full.sh --full on a fresh pod ≈ 50 min unsupervised
  • HF Xet backend handles the big model downloads (gigabytes per second on Pro tier)
  • End of run: PuLID-FLUX import smoke-test, HF buckets created (lhfam-models, lhfam-outputs), all five venvs verified
  • Pinned to development-timothy branch of LHFAM repo

backup story (HF buckets)

HuggingFace Pro ($9/mo) gives 1TB private storage with Xet content-addressable dedup. We push the entire /workspace/{PuLID,echomimic_v3,Wan2.2-S2V-14B,InfiniteYou,hf_cache} set to a private bucket with one command:

  • scripts/hf_backup.sh push-models — mirror models to bucket
  • scripts/hf_backup.sh push-outputs — append outputs (no --delete)
  • scripts/hf_backup.sh restore-all — disaster recovery onto a new pod

Today's first push: 67.5GB in ~7 minutes. Re-pushes are near-instant via Xet dedup. This means we can lose any pod (or stop one to save GPU $) and rebuild from canonical bucket state in ~10-15 min instead of re-downloading from public HF.

Why this matters: the May 4-5 work was bottlenecked by 8+ hours of cumulative model downloading every time a pod broke. Today's installer + HF buckets reduces that to a 10-15 min restore. We can now iterate on GPU sizing (4090 ↔ H100 ↔ H200) without losing a half-day to setup each time.

7. what's next

Concrete next moves, in priority order. Most are 1-2 hours of GPU each.

  1. Real EchoMimic vs Wan2.2-S2V evaluation. Take 5 expressions from the new bible × 5 audio clips of varying energy → 25 talking-head clips per model = 50 total. Same audio across both models for direct A/B. Total GPU: ~50 clips × 6 min avg = 5 hours, ~$15.
  2. Higher-resolution Wan2.2-S2V outputs. Today's runs were the 576×576 default (square input). Wan2.2 supports 832×480, 1280×720, 1280×704. Worth burning a few clips at 1280×720 to see if quality justifies the longer render time.
  3. Test the full pipeline end-to-end on the new stack. Source clip → reaction line → Chatterbox audio → Wan2.2-S2V talking head → ffmpeg PiP splice. May 5 had this for EchoMimic; we should re-run it on Wan2.2-S2V output to see if PiP composition reveals any new artifacts.
  4. InfiniteYou as identity injector. Pre-installed but not yet tested. Per ICCV 2025 it has 80% win rate against PuLID-FLUX for identity preservation. Easy 1-hour experiment: same prompt grid as the bible but with InfiniteYou instead of PuLID.
  5. Wan2.2-Animate-14B (character replacement on existing video). Different beast — takes a source video + reference photo, swaps the person. Could be the path to "Mason in a real creator-room setup" without needing to compose backgrounds ourselves.
What we should NOT do until D1-D6 (from the May-5 page) are settled: production-scale batch generation. Today's run validates the H100 stack; D1-D6 still need the team to weigh in on character direction, voice strategy, aesthetic, and quality bar before we light up the pod for an 8-hour batch.
